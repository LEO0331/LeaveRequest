# Leave Management System 系統設計審查（繁體中文）

## 1) 範圍與系統脈絡
此專案是 React + TypeScript 的單頁應用（SPA），提供請假流程操作，具備兩種角色（`Employee`、`Manager`）。

功能範圍：
- 建立/編輯請假單
- 核准/拒絕/取消/刪除（依角色限制）
- 清單搜尋/篩選/排序/分頁
- CSV 匯入匯出（含逐列錯誤報告）
- PDF 匯出（含目前篩選條件摘要）
- 使用瀏覽器 `localStorage` 做本地持久化

非功能重點：
- 不依賴後端往返，維持本地互動速度
- 強化前端驗證一致性
- UI 組件拆分，便於維護

## 2) 架構總覽
### 執行拓樸
- 純前端執行（Runtime 沒有 API 依賴）
- 持久化：瀏覽器 `localStorage`
- 建置/部署：Vite + GitHub Actions + GitHub Pages

### 模組分工
- `src/App.tsx`：編排層與主狀態擁有者
- `src/components/*`：UI 呈現與互動元件
- `src/lib/storage.ts`：資料載入/儲存/種子資料
- `src/lib/validation.ts`：商業規則驗證
- `src/lib/date.ts`：工時計算與時間轉換
- `src/lib/csv.ts`：CSV 解析/序列化/防護
- `src/lib/pdf.ts`：延遲載入 PDF 匯出
- `src/lib/app-layer.ts`：排序、對話框文案、摘要、歷程附加等工具

### 寫入資料流
1. 使用者操作觸發表單/對話框（`App.tsx`）
2. 呼叫 `validateDraft(...)` 進行規則檢查
3. 用 `toLeaveRequest(...)` 正規化資料
4. 更新記憶體狀態陣列（`setRequests`）
5. 透過 `saveLeaveRequests(...)` 寫入 `localStorage`
6. 以 Snackbar 顯示成功/失敗回饋

### 查詢資料流
1. 以 request 陣列作為資料來源
2. 使用 `useMemo` 產生衍生結果：篩選 -> 排序 -> 分頁切片
3. 由 `RequestTable` 呈現
4. 列選取帶動 `DetailsDrawer`

## 3) 關鍵設計決策與取捨
### 決策 A：純前端架構（無後端）
選擇原因：
- 部署與維運成本低
- 互動延遲低
- 產品驗證迭代快

取捨：
- 跨裝置不同步
- 無集中式權限與稽核
- 受限於瀏覽器儲存容量與可靠性

替代方案：
- API + 資料庫（Postgres / DynamoDB 等）
適用情境：
- 多人協作、法遵要求、組織級報表

### 決策 B：`App.tsx` 單一狀態來源
選擇原因：
- 資料流可預測、除錯容易
- 避免過早引入全域狀態工具

取捨：
- 頂層元件體積較大
- props 傳遞層級增加

替代方案：
- `Redux Toolkit`、`Zustand` 或 Context 分域狀態
適用情境：
- 多頁共享狀態、即時更新場景更多

### 決策 C：`src/lib/*` 的邏輯邊界
選擇原因：
- 將商業邏輯與 UI 呈現解耦
- 可針對邏輯模組直接做單元測試

取捨：
- 模組間可能出現部分轉換邏輯重疊

替代方案：
- 完整領域服務層（DDD 風格）
適用情境：
- 工作流程更複雜、團隊規模更大

## 4) 資料結構：為何這樣選、替代方案與取捨

### 4.1 主要集合使用 `LeaveRequest[]`
現況：
- 記憶體主資料結構為陣列，用於篩選、排序、分頁與渲染

選擇原因：
- 對表格「有序呈現」非常直觀
- 不可變更新語義簡單
- 與 `.filter/.map/.slice/.sort` 高度契合

取捨：
- 依 id 查找為 O(n)
- 驗證與聚合常需重複掃描

替代方案：
1. `Map<string, LeaveRequest>` + 額外排序 id 清單
- 優點：依 id 查找/更新可 O(1)
- 缺點：排序、篩選、分頁流程更複雜

2. 正規化結構 `{ byId, allIds }`
- 優點：可擴展的更新模式
- 缺點：小型專案會有較多樣板負擔

### 4.2 `Map` 用於分組與 CSV upsert
現況：
- 分組摘要：`Map<client, count>`
- CSV 匯入合併：`Map<id, LeaveRequest>`

選擇原因：
- 平均 O(1) key 更新
- 避免巢狀迴圈造成 upsert 效能劣化

替代方案：
1. 一般物件字典（Object）
- 優點：語法更簡單
- 缺點：key 語意與 prototype 風險較差

2. 陣列 reduce + `find`
- 優點：不需額外結構
- 缺點：upsert 易變 O(n^2)

### 4.3 角色/狀態/行為使用字串聯集（Union）
現況：
- `ActorRole`、`LeaveStatus`、`LeaveType`、`ActionType` 都是 TS 字串聯集

選擇原因：
- 編譯期型別安全
- 條件分支有完整性提示

替代方案：
1. TypeScript `enum`
- 優點：命名空間更明確
- 缺點：有 runtime 輸出，與字串互操作性較差

2. 全部用裸字串
- 優點：寫法最少
- 缺點：容易打錯字、重構風險高

### 4.4 `LeaveAuditEntry[]` 追加式歷程
現況：
- 每張請假單保留操作歷程（動作、角色、時間、備註）

選擇原因：
- 支援基本稽核可追溯性
- 本地不可變 append 容易實作

取捨：
- 歷程成長會增加單筆 payload

替代方案：
1. 全域事件日誌（依 request id 關聯）
- 優點：分析與查詢彈性高
- 缺點：明細畫面需要 join

2. 只留最後一次動作
- 優點：儲存最精簡
- 缺點：失去完整稽核軌跡

### 4.5 日期欄位採 ISO 字串
現況：
- 時間欄位以 ISO 儲存

選擇原因：
- 對 `localStorage` / CSV / PDF 序列化穩定
- 可被 `Date` 直接解析

取捨：
- 重複解析有成本
- 顯示受本地時區/locale 影響

替代方案：
1. Epoch 毫秒（number）
- 優點：比較運算快
- 缺點：除錯與匯出可讀性較差

2. 僅日期格式（`YYYY-MM-DD`）+ 時段欄位
- 優點：更貼近工作日規則語意
- 缺點：對部分時數請假彈性較差

## 5) 驗證與一致性策略
現況：
- `validateDraft(...)` 做同步驗證
- 重疊檢查排除 `Cancelled` / `Rejected`
- 配額占用只計 `Submitted` / `Approved`
- 工時計算排除週末與公司假日

優點：
- 規則可預測、即時回饋
- 驗證邏輯集中，不依賴 UI 控件狀態

取捨：
- 純前端模式下，完整資料完整性無法強制

替代方案：
- 後端權威驗證 + 簽章式稽核軌跡

## 6) 效能與可擴展性
現況：
- 種子資料量大（10k），但有分頁
- 主要衍生運算使用 `useMemo`
- PDF 套件延遲載入，降低初始 bundle 壓力

進一步擴展可能瓶頸：
- 每次依賴變動都需重新篩選/排序（O(n log n)）
- 聚合統計需全量掃描

替代策略：
1. 常用查詢索引化（status/user/date）
2. 表格虛擬化（windowing）
3. 查詢下放後端

## 7) 安全性與可靠性
已實作控制：
- CSV 匯出防公式注入
- 匯入檔案副檔名與大小限制
- 角色導向 UI 動作限制

設計上的缺口（client-only 本質）：
- 角色控制不是強安全邊界
- 本地資料可被外部工具改寫

## 8) Deep Dive 問答準備
### Q1：為何現在不直接上資料庫？
A：目前目標是本地流程驗證與快速交付；client-only 可降低維運成本並提高迭代速度。若進入多人協作與法遵需求，後端是下一步。

### Q2：為何不用 normalized store？
A：此專案主要負載是表格排序/篩選/分頁，陣列可讀性高且直觀。Map/normalized 結構雖有更新優勢，但在目前規模會增加複雜度。

### Q3：如何處理並發衝突？
A：目前沒有跨客戶端並發問題（資料在本機）。若遷移後端，建議加上 optimistic concurrency（版本號/ETag）。

### Q4：為何稽核歷程放在每筆 request 內？
A：明細抽屜需要快速顯示本筆時間線，放在同筆資料最直接。若未來要做全域分析，應改為獨立事件儲存。

### Q5：完整性邊界在哪裡？
A：完整性邊界在 `validateDraft` 與 CSV parser。安全邊界在 client-only 模式下較弱，真正強制需 server-side policy。

### Q6：擴展時優先重構點？
A：
1. 將 `App.tsx` 中的領域流程抽出 service 層
2. 對常用過濾建立索引
3. 表格導入虛擬化
4. 改為 API + DB 並將驗證上移到後端

### Q7：為何 PDF 套件要延遲載入？
A：PDF 是次要操作，延遲載入可縮短初始載入路徑並改善 Lighthouse 指標。

### Q8：為何 CSV 匯入合併用 `Map`？
A：以 id upsert 可達 `O(n + m)`，避免巢狀掃描 `O(n*m)`，最後再轉回陣列供 UI 呈現。

## 9) 建議的下一步架構里程碑
1. 導入後端 API，將角色與驗證權威化
2. 稽核事件改為後端 immutable append
3. 大資料量改為後端查詢/分頁
4. 增加可觀測性（匯入/匯出失敗的結構化記錄與指標）
